# Publishing

Status: draft
Last updated: 2026-05-06
Target version: v0.2.4

## Goal

Writers can publish the active Markdown document to one of four blog
platforms — Ghost, WordPress (self-hosted), WordPress.com, or Medium —
without leaving Novelist. Each publish action targets exactly one
configured channel; metadata is entered in a dialog at publish time.

This spec covers publish channels only. The image-host feature is in a
separate v0.2.4 spec; the two interact at the "upload local images"
step.

## Non-Goals

- Update an already-published post. Every publish creates a new post.
  Re-publishing an edited document creates duplicates that the user
  prunes manually on the platform side.
- Cross-posting in one action. One publish action targets one channel.
  Cross-posting requires N separate user actions.
- YAML front-matter parsing or writeback. The dialog is the single
  source of metadata input. Any front-matter present in the doc is
  stripped from the body before submission and otherwise ignored.
- Browser-redirect auth flows. Manual paste of credentials only.
  Application Password browser flow (WP) and OAuth2 (WP.com) are
  deferred to a later release.
- Scheduled publishing UI. Status dropdown does not surface
  `scheduled` / `future`. Users who want scheduling can switch
  status on the platform's web UI after the post lands as a draft.
- Front-matter ID tracking. No sidecar file or YAML key remembers
  which post a doc maps to on each platform.

## Platforms

Four user-visible platforms; five distinct Rust adapter modules
(WP self-hosted and WP.com share enough wire shape that they could
collapse into one file, but stay as separate adapters for clarity).
Each adapter is a self-contained Rust function — no shared trait —
mirroring the image-host architecture from the sister spec.

| Platform | Auth | Body format | Media upload |
|---|---|---|---|
| Ghost | Admin API key (`id:secret` hex) signed as 5-min HS256 JWT, `Authorization: Ghost <token>`, `Accept-Version: v5.0` | HTML (`?source=html` URL param converts to Lexical server-side) | `POST /ghost/api/admin/images/upload/` (multipart) |
| WordPress self-hosted | HTTP Basic with Application Password (WP 5.6+); username + 24-char password | HTML | `POST /wp-json/wp/v2/media` (multipart or raw with `Content-Disposition`) |
| WordPress.com | Bearer access token | HTML | `POST https://public-api.wordpress.com/wp/v2/sites/<id>/media` |
| Medium | Bearer integration token | Markdown (native) | `POST https://api.medium.com/v1/images` (multipart) |

Note on Medium: the Integration Token UI was removed from Medium's
account settings in late 2024 per multiple 2024-2025 reports. Endpoints
still accept legacy tokens. New users typically cannot generate a
token; the feature ships anyway for users who already hold one.

## Configuration

Plaintext credentials in global settings only (mirrors image-host).
Per-project overrides not supported for v0.2.4.

```
GlobalSettings.publish:
  channels: [
    { id: <uuid>, name: "Personal Ghost", platform: "ghost", admin_url: "...", api_key: "..." },
    { id: <uuid>, name: "Company WP",     platform: "wordpress_self_hosted", site_url: "...", username: "...", app_password: "..." },
    { id: <uuid>, name: "WP.com Blog",    platform: "wordpress_com", site_id_or_domain: "myblog.wordpress.com", access_token: "..." },
    { id: <uuid>, name: "Medium",         platform: "medium", token: "..." },
  ]
```

No "active channel" pointer — the user picks a channel per publish
action via the share menu.

## User Behaviors

### Configuring channels (Settings → Publish)

- Add channel: pick platform, fill platform-specific form fields, save.
- Edit / Delete / Test buttons per channel.
- Test: per-platform "verify credentials" call — Ghost and WP issue a
  `GET /users/me`-style call; Medium issues `GET /v1/me`. No post is
  created. Reports inline pass/fail with the platform's error message.

### Editor share button (new top-right toolbar)

A single dropdown button in the editor header. Items:

- Copy HTML
- Export as… (delegates to existing export dialog)
- Upload Local Images to… (delegates to v0.2.4 image-host palette command)
- ─── divider ───
- For each configured channel: `Publish to <name>`

Each `Publish to <name>` opens the publish dialog scoped to that
channel.

### Publish dialog (per-channel)

Header: channel display name and base URL (e.g.
`Personal Ghost — https://blog.example.com`).

Fields:

- **Title** — pre-filled from the document's first H1 (or filename if
  no H1). Editable. Required.
- **Tags** — chip input. Comma or Enter to add. Removable.
- **Excerpt** — optional textarea. Empty = platform's default behavior.
- **Slug** — auto-generated kebab-case from title. Editable.
- **Cover image** — drag-drop or pick. Optional. The selected image is
  uploaded to the platform's media endpoint as part of pre-publish.
- **Status dropdown** — per-platform options (see below).
- **Cancel** / **Publish** buttons in footer.

Per-platform status options:

| Platform | Status options | Default |
|---|---|---|
| Ghost | `draft`, `published` | `draft` |
| WP self-hosted | `draft`, `pending`, `private`, `publish` | `draft` |
| WP.com | `draft`, `pending`, `private`, `publish` | `draft` |
| Medium | `draft`, `unlisted`, `public` | `public` |

### Pre-publish image rewrite

On Publish click, the orchestrator:

1. Strips any YAML front-matter from the body.
2. Enumerates `![alt](path)` references whose target is not an
   absolute URL.
3. For each local reference, calls
   `upload_post_image_<platform>(bytes, ...)` to upload to the publish
   target's own media endpoint, returning the platform-hosted URL.
4. Rewrites the body, replacing each local reference with the new URL.
5. For Ghost / WP / WP.com: converts the rewritten Markdown to HTML
   via Pandoc (already shipped with Novelist).
6. For Medium: passes the rewritten Markdown unchanged with
   `contentFormat: "markdown"`.
7. Submits the create-post request to the platform.

If any image upload step fails, the publish aborts and the dialog
shows an inline error identifying the failed image. The user can
retry without re-entering metadata.

If the user wants images hosted on a third-party CDN instead of the
publish platform, they explicitly run the `Upload Local Images to…`
share-menu action first (which uses the configured image host); the
publish step then sees no local references and skips upload.

### Post-publish surface

On success, a toast appears with the platform URL and an "Open in
browser" action that launches the user's default browser.

On failure, the dialog stays open with an inline error message
(network / auth / quota / server-error / unparseable-response /
bad-config). User can edit fields and retry without closing.

## Acceptance Criteria

- A user with valid Ghost / WP / WP.com / Medium credentials can
  configure a channel, set its name, and successfully publish the
  active Markdown doc.
- Local image references in the doc are uploaded to the platform's
  media endpoint and replaced with platform-hosted URLs in the
  submitted body.
- Failed image upload aborts publish without partial submission.
- Ghost JWT auth is generated fresh per request with a 5-minute TTL.
- WordPress 401 on bad Application Password produces a clear error
  message that names the platform's response text.
- The publish dialog can be cancelled at any point without leaving
  the document in a modified state (no body changes are persisted to
  disk during pre-publish — the rewrite is in-memory only and
  discarded on cancel).
- The CodeMirror document buffer is never modified by publish
  operations. Local image references stay local in the user's source.

## Test Harness

- **Rust unit tests** (`core/src/services/publish/*.rs`): one per
  platform, `wiremock`-stubbed. Cases: success, 401, 429, 5xx,
  malformed response, bad config. Ghost: HS256 JWT signing has a
  known-vector test verifying header/payload/signature against fixed
  inputs.
- **Pandoc helper test**: integration test invoking
  `convert_markdown_to_html` on representative inputs (paragraph,
  heading, code block, image, CJK paragraph). Skipped on systems
  without Pandoc binary; logs a warning.
- **Frontend unit tests** (`tests/unit/services/publish.test.ts`):
  mocked `invoke`, verifies platform routing, image rewrite step,
  body-format selection, retry on failure preserves dialog state.
- **Settings UI** (`tests/unit/components/settings-publish.test.ts`):
  add/edit/delete a channel, test button calls the right command.
- **Manual live-API verification** before release: opt-in checklist
  (not in CI) — one real publish to each platform with valid
  credentials, then delete the resulting drafts.
